# ADR-0005: build.rs 与 Proto 代码生成

- Status: Accepted
- Date: 2026-04-10
- Issue: #725

## Context

`koduck-ai` 在 Task 2.1 中已冻结了 memory/tool/llm 三个服务的 proto 契约（ADR-0004），`build.rs` 已能通过 `tonic-build` 编译全部 4 个 proto 文件到 `OUT_DIR`。但目前生成的类型没有通过 Rust 模块暴露给业务代码，orchestrator 无法直接引用 memory/tool/llm 的 client trait。

### 问题

1. **生成的代码不可访问**：`tonic-build` 将 Rust 代码输出到 `OUT_DIR`，但没有 `mod.rs` 使用 `tonic::include_proto!` 宏将其引入模块树
2. **client trait 无法注入**：orchestrator 需要依赖 `memory_service_client::MemoryServiceClient<T>` 等 tonic 生成的 client 类型，但当前这些类型不在任何公开模块中
3. **跨包引用**：memory/tool/llm proto 文件 import 了 `shared.proto`，生成的代码使用 `super::super::contract::v1::RequestMeta` 等相对路径引用 contract 类型，需要与 Rust 模块结构匹配

### 现有参考

- `koduck-auth/src/grpc/proto/mod.rs`：使用 `tonic::include_proto!("koduck.auth.v1")` 和 `tonic::include_file_descriptor_set!("koduck")` 暴露生成代码
- `koduck-auth/build.rs`：单 proto 文件编译，`file_descriptor_set_path` 为 `koduck.bin`

## Decision

### 1. 两阶段编译 + extern_path

核心挑战是 `tonic::include_proto!` 将生成的类型直接放在调用模块级别（不创建子模块），且跨包引用使用相对路径 `super::super::contract::v1::RequestMeta`，与 Rust 模块结构不匹配。

采用两阶段编译策略：

```rust
// build.rs

// 阶段 1：编译 shared.proto（无 extern_path），生成 contract 类型
tonic_build::configure()
    .compile(&["proto/koduck/contract/v1/shared.proto"], &["proto", "/usr/include"])?;

// 阶段 2：编译 service protos，使用 extern_path 将 contract 引用指向绝对路径
tonic_build::configure()
    .extern_path(".koduck.contract.v1", "crate::clients::proto")
    .compile(&[
        "proto/koduck/memory/v1/memory.proto",
        "proto/koduck/tool/v1/tool.proto",
        "proto/koduck/llm/v1/llm.proto",
    ], &["proto", "/usr/include"])?;
```

`extern_path` 告诉 tonic-build：不要为 `koduck.contract.v1` 生成代码，服务 proto 中的 contract 类型引用使用绝对路径 `crate::clients::proto::RequestMeta` 替代相对路径。

### 2. Proto 模块结构

```
src/clients/
  proto/
    mod.rs          # 所有 tonic::include_proto! + re-export + file_descriptor_set
  memory/
    mod.rs          # re-export MemoryServiceClient 等
  tool/
    mod.rs          # re-export ToolServiceClient 等
  llm/
    mod.rs          # re-export LlmServiceClient 等
  mod.rs            # pub mod proto; pub mod memory; pub mod tool; pub mod llm;
```

### 3. proto/mod.rs 内容

```rust
// Contract types - generated by pass 1
tonic::include_proto!("koduck.contract.v1");

// Service types - generated by pass 2 (extern_path resolves contract references)
tonic::include_proto!("koduck.memory.v1");
tonic::include_proto!("koduck.tool.v1");
tonic::include_proto!("koduck.llm.v1");

// Re-export key client/server traits
pub use memory_service_client::MemoryServiceClient;
pub use memory_service_server::{MemoryService, MemoryServiceServer};
pub use tool_service_client::ToolServiceClient;
pub use tool_service_server::{ToolService, ToolServiceServer};
pub use llm_service_client::LlmServiceClient;
pub use llm_service_server::{LlmService, LlmServiceServer};

pub const FILE_DESCRIPTOR_SET: &[u8] = tonic::include_file_descriptor_set!("koduck_ai");
```

所有 proto 类型放在同一模块中（`clients::proto`），client/server trait 在模块级别 re-export。

### 4. Client 模块 Re-export

```rust
// src/clients/memory/mod.rs
pub use super::proto::MemoryServiceClient;
pub use super::proto::{MemoryService, MemoryServiceServer};
```

Orchestrator 可通过 `clients::memory::MemoryServiceClient<T>` 或 `clients::proto::MemoryServiceClient` 直接使用。

### 5. CI 集成

- `build.rs` 中 `cargo:rerun-if-changed={proto}` 确保 proto 变更触发重新编译
- `docker build` 执行完整编译验证

## Consequences

### 正向影响

1. **类型可直接引用**：orchestrator 可直接 import `clients::proto::MemoryServiceClient` 等 client trait
2. **依赖注入友好**：client trait 可作为 generic bound 注入到 orchestrator，便于测试 mock
3. **proto 变更自动检测**：任何 proto 文件修改都会触发 Cargo 重新编译
4. **跨包引用正确**：`extern_path` 确保生成的代码使用正确的绝对路径引用 contract 类型

### 代价与风险

1. **两阶段编译**：`build.rs` 复杂度增加，需要两次 `tonic_build::configure().compile()` 调用
2. **proto 变更触发全量重编译**：不可避免
3. **模块命名**：`tonic::include_proto!` 将类型放在模块级别而非子模块，同名类型可能冲突（当前 proto 设计已避免）

### 兼容性影响

- **API 兼容性**：纯新增模块，不影响已有代码
- **构建系统**：`build.rs` 变更为两阶段编译
- **下游服务**：无影响，proto 契约不变

## Alternatives Considered

### 1. 单次编译 + 模块结构匹配相对路径

- **方案**：通过 `contract/v1/mod.rs` 等子模块结构匹配生成的 `super::super::contract::v1` 相对路径
- **拒绝理由**：`tonic::include_proto!` 创建的模块通过 `include!()` 引入，无法从其他文件访问（模块可见性限制）

### 2. 将生成代码写入 src/ 目录（而非 OUT_DIR）

- **方案**：使用 `tonic_build::configure().out_dir("src/clients/proto")` 将生成文件写入源码目录
- **拒绝理由**：生成代码混入源码目录，增加 git 管理复杂度；`OUT_DIR` + `include_proto!` 是 tonic 推荐的标准做法

### 3. 使用 buf 构建 proto 而非 tonic-build

- **优点**：buf 提供 lint、format、breaking change detection 等工具链
- **拒绝理由**：引入额外工具依赖，当前阶段 tonic-build 已满足需求

## Verification

- `docker build -t koduck-ai:dev ./koduck-ai` 编译通过
- `clients::proto::MemoryServiceClient` 类型可引用
- `clients::proto::RequestMeta` 类型可引用
- `clients::memory::MemoryServiceClient` re-export 可用
- 修改任意 proto 文件后 `cargo build` 自动重新编译
