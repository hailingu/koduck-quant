# ADR-0002: 测试覆盖率门禁机制

## 元数据

- **状态**: 已接受
- **日期**: 2026-03-31
- **作者**: @hailingu
- **相关**: #115, #73, docs/phase3/coverage-gate-plan.md

---

## 背景与问题陈述

随着 koduck-quant 项目代码量增长，测试覆盖率下降，导致：
- 代码质量难以保证
- 重构缺乏信心
- 回归缺陷增加

需要建立测试覆盖率门禁机制，确保代码质量。

### 上下文

- **当前覆盖率**: Java 后端约 45%
- **目标**: 业务逻辑 ≥ 80%，核心路径 100%
- **团队现状**: 测试意识不足，缺乏强制执行机制

---

## 决策驱动因素

1. **质量保证**: 通过自动化门禁防止覆盖率下降
2. **可维护性**: 高覆盖率代码更容易维护和重构
3. **团队习惯**: 培养编写测试的习惯
4. **实施成本**: 门禁配置和运维成本

---

## 考虑的选项

### 选项 1: CI 硬性门禁（推荐）

**描述**: CI 中配置 JaCoCo 覆盖率检查，不达标则阻断 PR

**优点**:
- 强制性强，无法绕过
- 自动化，无需人工干预
- 即时反馈

**缺点**:
- 初期可能导致 PR 延迟
- 需要处理遗留代码覆盖率

### 选项 2: 软性提醒（Warning）

**描述**: CI 显示覆盖率报告和警告，但不阻断 PR

**优点**:
- 不会阻塞开发流程
- 渐进式改进

**缺点**:
- 容易被忽视
- 效果依赖团队自觉性

### 选项 3: 定期审计

**描述**: 定期（如每月）人工审查覆盖率报告

**优点**:
- 灵活，可根据情况调整
- 可以结合代码质量综合评估

**缺点**:
- 人工成本高
- 反馈不及时

---

## 决策结果

**选定的方案**: 选项 1 - CI 硬性门禁（分阶段实施）

**理由**:

1. **有效性**: 硬性门禁是确保覆盖率提升的最有效手段
2. **分阶段实施**: 避免对现有开发流程造成过大冲击
   - Phase 1: 60% 门禁（当前可达）
   - Phase 2: 70% 门禁
   - Phase 3: 80% 门禁（目标）

**积极后果**:

- 测试覆盖率持续提升
- 代码质量改善
- 重构信心增强

**消极后果**:

- 短期内开发效率可能受影响
- 需要补充遗留代码测试

**缓解措施**:

- 遗留代码可以标记 `@Generated` 排除
- 提供测试编写培训和模板
- 设置合理的过渡期

---

## 实施细节

### 门禁配置

```xml
<!-- pom.xml -->
<plugin>
    <groupId>org.jacoco</groupId>
    <artifactId>jacoco-maven-plugin</artifactId>
    <configuration>
        <rules>
            <rule>
                <element>BUNDLE</element>
                <limits>
                    <limit>
                        <counter>INSTRUCTION</counter>
                        <value>COVEREDRATIO</value>
                        <minimum>0.60</minimum>
                    </limit>
                </limits>
            </rule>
        </rules>
    </configuration>
</plugin>
```

### 分阶段目标

| 阶段 | 时间 | 门禁阈值 | 说明 |
|------|------|----------|------|
| Phase 1 | 2026-04 | 60% | 当前水平，稳定 CI |
| Phase 2 | 2026-05 | 70% | 逐步提升 |
| Phase 3 | 2026-06 | 80% | 达到目标 |

### 例外处理

- 纯配置类、常量类可排除
- 遗留代码可临时标记排除（需创建 Tech Debt Issue）
