# ADR-1001: WebSocket 实时数据推送架构

## 元数据

- **状态**: 已接受

- **相关**: #271, issues/websocket-backend-infrastructure.md

---

## 背景与问题陈述

koduck-quant 需要提供股票实时价格推送功能，让前端能够及时获取市场数据更新。我们需要选择一种合适的实时通信技术方案。

### 上下文

- **业务需求**: 用户需要实时查看股票价格变动，延迟应控制在秒级
- **技术背景**: 当前系统使用 REST API 提供数据，无法支持服务器主动推送
- **用户规模**: 预计同时在线用户 1000+，单用户关注股票 10-50 只

---

## 决策驱动因素

1. **实时性**: 价格更新需要在 1-3 秒内推送到客户端
2. **可扩展性**: 支持水平扩展，应对用户增长
3. **兼容性**: 支持主流浏览器和移动设备
4. **开发效率**: 团队熟悉度，学习成本
5. **运维复杂度**: 部署和维护成本

---

## 考虑的选项

### 选项 1: WebSocket + STOMP

**描述**: 使用 Spring Boot WebSocket 支持，配合 STOMP 协议进行消息路由

**优点**:
- 全双工通信，真正的实时推送
- STOMP 提供消息路由、订阅管理
- Spring Boot 原生支持，集成简单
- 支持 SockJS fallback

**缺点**:
- 需要维护长连接状态
- 水平扩展需要共享会话状态（Redis）
- 需要处理连接异常、心跳、重连等

### 选项 2: Server-Sent Events (SSE)

**描述**: 使用 HTTP 协议的服务器推送技术

**优点**:
- 基于 HTTP，穿透性好，无需特殊代理配置
- 自动重连机制
- 实现简单，客户端使用 EventSource API

**缺点**:
- 单向通信（服务器 → 客户端）
- 浏览器连接数限制（6 个/域名）
- 不支持二进制数据

### 选项 3: 轮询 (Polling)

**描述**: 客户端定时请求最新数据

**优点**:
- 实现最简单
- 兼容所有环境
- 无状态，水平扩展容易

**缺点**:
- 实时性差（取决于轮询间隔）
- 无效请求多，浪费资源
- 服务器压力大

### 选项 4: 长轮询 (Long Polling)

**描述**: 改进的轮询，服务器保持请求直到有数据

**优点**:
- 比轮询实时性好
- 基于 HTTP，兼容性好

**缺点**:
- 实现复杂
- 连接频繁建立/断开
- 服务器资源消耗大

---

## 决策结果

**选定的方案**: 选项 1 - WebSocket + STOMP

**理由**:

1. **实时性最佳**: WebSocket 提供真正的双向实时通信，满足股票推送的实时性要求
2. **技术栈匹配**: Spring Boot 对 WebSocket 提供优秀支持，团队有使用经验
3. **功能完整**: STOMP 提供订阅管理、消息路由，满足多股票订阅场景
4. **扩展性**: 配合 Redis 可以实现分布式消息广播

**积极后果**:

- 用户能够实时接收价格更新
- 可以支持更复杂的交互（如用户订阅特定股票）
- 为后续实时功能（如预警推送）奠定基础

**消极后果**:

- 需要引入 Redis 管理分布式会话
- 需要处理网络不稳定场景的重连
- 运维复杂度增加

**缓解措施**:

- 实施心跳检测和自动重连机制
- 使用 SockJS 提供 fallback 支持
- 详细记录运维手册

## 实施细节

### 技术栈

- **后端**: Spring Boot WebSocket + STOMP
- **消息代理**: Redis Pub/Sub（分布式场景）
- **前端**: @stomp/stompjs + ws
- **协议**: STOMP over WebSocket
