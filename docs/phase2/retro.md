## 5. 经验教训

### 5.1 技术层面

1. **Maven 配置冲突**: 多个 feature 同时修改 pom.xml 容易产生冲突
   - *教训*: 尽早合并基础配置，或使用 BOM 管理依赖

2. **测试数据工厂**: TestDataFactory 模式大幅提升测试编写效率
   - *经验*: 统一测试数据构造可减少重复代码

3. **worktree 并行开发**: 有效避免分支切换的上下文丢失
   - *经验*: 适合独立且并行的任务

### 5.2 流程层面

1. **依赖分析前置**: 提前画出任务依赖图，避免返工
   - *经验*: 复杂项目必须先分析依赖关系

2. **模板化**: PR/Issue 模板强制规范输入，减少沟通成本
   - *经验*: 模板要包含 DoD 和检查清单

3. **自动化**: 一键脚本比文档更有用
   - *经验*: 开发者更倾向于运行脚本而不是阅读文档

## 7. 团队建议

### 7.1 开发规范

1. **提交前强制检查**: 建议配置 git pre-commit hook 运行 `make quality`
2. **测试覆盖率门禁**: PR 必须达到 60% 覆盖率
3. **Code Review 清单**: PR 审查时使用标准化检查清单

### 7.2 工具建议

1. **IDE 插件**: 推荐安装 SonarLint 插件，实时代码质量提示
2. **本地数据库**: 使用 Testcontainers 替代 H2，更接近生产环境
3. **性能测试**: 建立每周性能回归测试机制

### 7.3 流程优化

1. **Sprint 计划**: 预留 20% 缓冲时间处理突发问题
2. **每日站会**: 同步 Phase 3 进度和阻塞问题
3. **周会复盘**: 每周回顾质量和进度指标
